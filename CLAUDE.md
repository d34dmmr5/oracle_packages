# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## О проекте

Репозиторий содержит PL/SQL-исходники для Oracle-схемы `BA7_DATA` — учётной системы объектов
недвижимости. «Объект» (`OBJECT`) — это универсальная сущность с подтипами: дом (`OBJECT_HOUSE`),
подъезд (`HOUSE_DOORWAY`), помещение/квартира (`OBJECT_FLAT`), комната (`OBJECT_ROOM`), «прочий»
объект (`OBJECT_UNKNOWN`). Адрес объекта разрешается через иерархический справочник территорий
(`TERRITORY`): страна → регион → район → населённый пункт → внутригородской район → улица.

Сборки/тестов/CI в репозитории нет — это чистый набор SQL/PLSQL-скриптов для ручного разворачивания
через SQL*Plus (или другой Oracle-клиент) под пользователем `ba7_data`. «Проверка» изменений на
практике — это компиляция объекта в реальной Oracle-схеме (`CREATE OR REPLACE ... / show errors`) и
ручные `SELECT` из `object_info`.

Сама таблица `OBJECT`/`TERRITORY` и другие базовые таблицы в репозитории не создаются (DDL для них
здесь нет) — есть только `ALTER TABLE` для `object_info_tbl` и процедурная логика (пакеты, триггеры).

## Структура репозитория

- **`territory_pkg.sql` / `territory_pkg.body.sql`** — пакет `territory_pkg`, процедура `get_info`.
  Обходит иерархию `TERRITORY` вверх от заданной территории (`CONNECT BY ... START WITH ...
  id_parent`), строит `short_adres`/`full_adres`, распознаёt класс каждого предка
  (`id_territory_class`: `8`=улица, `5`=внутригородской район, `4`=населённый пункт, `7`=район,
  `3`=регион, `2`=страна), считает `adr_pos01..10` — позиции символов в `full_adres`, с которых
  начинается адрес соответствующего уровня (чтобы делать `SUBSTR` без повторной сборки строки), и
  почтовый индекс.
- **`GetTerritoryInfo.sql`** — тонкая обёртка над `territory_pkg.get_info`, отдаёт наружу только
  подмножество out-параметров, нужное `object_info_pkg` (без `*_type`-полей). В конце файла —
  регистрация синонимов/прав через `scheming_pkg` для схем `TEST_OWNER`/`STD_POLICY`.
- **`object_info_pkg.sql`** — центральный, актуальный пакет проекта. Поддерживает денормализованную
  отчётную таблицу `object_info_tbl` (и view `object_info` над ней) в синхронизации с
  `OBJECT`/`TERRITORY`. Внутри:
  - Пакетные коллекции `l$id_object_insert/update/delete`, `l$id_territory_update`, наполняемые
    построчными (`FOR EACH ROW`) триггерами через `add_object_insert/update/delete` и
    `add_territory_update`, а затем разово обрабатываемые в `update_object_info()` на уровне
    STATEMENT-триггера — обход `ORA-04091` (mutating table) вместо compound trigger.
  - `create_triggers()` — генерирует сами триггеры на `OBJECT` и `TERRITORY` через
    `ddl_pkg.create_trigger` (метапрограммирование DDL, а не статичный `CREATE TRIGGER`).
  - `create_view(p$with_grants)` — пересоздаёт view `object_info` (+ опционально выдаёт права).
  - `rebuild()` / `rebuild(p$batch_size)` — полная пересборка `object_info_tbl`: удаляет
    «осиротевшие» строки, вставляет отсутствующие, затем прогоняет «пустые» `UPDATE` (`SET col =
    col`) по `object_house`/`object_flat`/`house_doorway`/`object_unknown`/`object_room`/`object`,
    чтобы форсировать пересчёт через существующие триггеры. Batch-версия использует `run_sql_parts`
    для обработки диапазонами `id_object`, чтобы не блокировать всю таблицу разом.
  - `format_object_no(p$object_no, p$numeric_len)` — детерминированная функция выравнивания
    цифровой части номера (дом/квартира/комната) под заданную длину; используется во всех
    `trigger.*.sql`.
  - `get_version()` — строка версии пакета; в теле функции — комментарий-changelog (v1..v10) с
    датами и авторами. При содержательных изменениях в пакете принято повышать эту версию и
    дописывать changelog.
  - `l$object_required_column_list` — список полей `OBJECT`, копируемых в `object_info_tbl`.
    Используется и для генерации тела триггера, и как источник для `update_object_info`. При
    добавлении нового синхронизируемого поля в `OBJECT` его нужно добавить сюда **и** обработать в
    `INSERT`/`UPDATE`-ветках `update_object_info(p$object OBJECT%ROWTYPE, p$action VARCHAR2)`.
- **`trigger.object_house.sql`, `trigger.object_flat.sql`, `trigger.house_doorway.sql`,
  `trigger.object_room.sql`, `trigger.object_unknown.sql`** — `BEFORE INSERT OR UPDATE FOR EACH ROW`
  триггеры на подтиповых таблицах. Их задача — скопировать/пересчитать адресные поля вниз, в общую
  `OBJECT`, и, где нужно, каскадно — на соседние подтиповые таблицы (см. ниже про `addressing_mode`).
- **`00_update_object_info_tbl.sql`, `01_procedure_triggers_views.sql`,
  `02_update_data_object_info_tbl.sql`** — **это не деплой-скрипты**, а хронологический лог
  изменений схемы (`ALTER TABLE` вперемешку с версиями старых триггеров, датами и инициалами
  авторов: PRA, SHA, ZAN). `01_procedure_triggers_views.sql` в самом начале прямо предупреждает: весь
  функционал актуализации `object_info` теперь находится в `object_info_pkg` (включая генерацию
  триггеров и view), а перечисленные там триггеры `TAIU6_OBJECT`, `TAIU6_TERRITORY`,
  `TAIU7_TERRITORY`, `TAIU1_OBJECT_FLAT`, `TAIU1_OBJECT_HOUSE` — устаревшие и заменены. Использовать
  эти 3 файла только чтобы понять историю/происхождение структуры `object_info_tbl`, не как
  источник актуального поведения.

## Ключевая доменная концепция: `addressing_mode`

`OBJECT_HOUSE.addressing_mode`:
- **`1` — «по домам»**: у дома единый адрес/номер, все его помещения, подъезды и комнаты наследуют
  номер дома напрямую от `OBJECT_HOUSE`.
- **`2` — «по подъездам»**: у дома нет единого номера — у каждого подъезда (`HOUSE_DOORWAY`) свой
  номер дома. Помещение обязано указывать `id_house_doorway` и наследует номер оттуда.

Каскады между таблицами:
- `trigger.object_house.sql`: при смене адресных полей/`addressing_mode` дома каскадно обновляет
  **все** его помещения, подъезды и комнаты (режим 1) либо помещения+комнаты через связку
  `object_flat → house_doorway` (режим 2).
- `trigger.house_doorway.sql`: обновляет `OBJECT` самого подъезда; в режиме 2, если сменился
  `house_no`, обновляет ещё и все помещения этого подъезда. Каскад ID здания в помещениях подъезда
  (при переносе подъезда на другое здание) сознательно вынесен на уровень STATEMENT-триггера
  (`tсbu1_house_doorway` / `tсau1_house_doorway`) через `num_collect_pkg` — тот же обход mutating
  table, что в `object_info_pkg`, но через именованные списки в сессии, а не пакетные переменные.
- `trigger.object_flat.sql`: помещение берёт адрес от дома (режим 1) либо от своего подъезда
  (режим 2); в режиме 2 требует заполненного `id_house_doorway`.
- `trigger.object_room.sql`: комната всегда наследует адрес от родительского помещения
  (`object_flat`), без развилки по `addressing_mode`.
- `trigger.object_unknown.sql`: простейший случай — копирует `id_territory`/`house_no`/`zip`
  напрямую в `OBJECT`, без подъездов/наследования.

Из кода видно, что `id_object_class = 10` — подъезд, `id_object_class = 11` — комната; `2`
встречается вместе с `10` в условии «ИД здания определено (помещение или подъезд)», что указывает на
`2` = помещение. Полного справочника классов в репозитории нет — при работе с конкретными ID
классов сверяться с реальной БД (`OBJECT_TYPE`), а не домысливать по коду.

## «Альтернативный адрес» (суффикс `2`)

Многие сущности имеют дублирующий набор полей с суффиксом `2` (`id_territory2`, `dom2`,
`building_no2`, `adres2`, `full_adres2`, ...) — независимый «второй» адрес объекта
(`is_exist_alternate_adres` в `object_info_tbl`). Он разрешается тем же кодом территорий, но никогда
не участвует в каскадах `addressing_mode` между подтиповыми таблицами — это данные только уровня
`OBJECT`/`object_info_tbl`.

## Технические особенности и подводные камни

1. **Кодировка файлов — Windows-1251 (cp1251), а не UTF-8**, кроме `object_info_pkg.sql` (он в
   UTF-8). Кириллица в остальных файлах при чтении инструментами, ожидающими UTF-8, превращается в
   мусор. Перед чтением/редактированием конвертировать: `iconv -f windows-1251 -t utf-8 файл.sql`.
   Перед сохранением изменений — конвертировать обратно (`iconv -f utf-8 -t windows-1251`), чтобы не
   сломать кодировку файла для остальных инструментов и Oracle-клиентов.
2. **Внешний «фреймворковый» слой не входит в репозиторий.** Код активно использует пакеты и типы,
   которых здесь нет: `scheming_pkg` (генерация имени схемы, синонимы/права для мульти-схемного
   деплоя между `TEST_OWNER`/`STD_POLICY`), `ddl_pkg` (генерация DDL триггеров),
   `int_rep_session` (управление режимом доступа к БД, обход репликации при `rebuild`),
   `num_collect_pkg` (именованные списки в сессии — обход mutating table в STATEMENT-триггерах),
   `string_pkg` (шаблонная подстановка для генерации кода), `run_sql_parts` (батчевое выполнение по
   диапазонам ID), типы `TTableNumber`/`TTableString`. Эти объекты нельзя редактировать в этом
   репозитории — они определены в общей инфраструктуре ba7-фреймворка за его пределами.
3. **«Пустой» `UPDATE ... SET col = col`** (в `00_`/`02_`-файлах и в `object_info_pkg.rebuild`) —
   не бессмысленная конструкция, а намеренный приём: форсировать срабатывание существующих
   BEFORE/AFTER-триггеров и пересчитать производные поля без изменения бизнес-данных.
4. **`GRANT`/`scheming_pkg`-хвосты** в конце большинства файлов (`group_synonym`, `group_privs` или
   прямые `GRANT`) — конвенция публикации нового объекта во все схемы-потребители
   (`TEST_OWNER(_ROLE)`, `STD_POLICY(_ROLE)`). При добавлении новых пакетов/view такие вызовы нужно
   добавлять по аналогии.
5. Комментарии `!!!` в коде — предупреждения авторов («устарело», «не нужен», «осторожно»),
   например в `01_procedure_triggers_views.sql` про `TAIU1_OBJECT_FLAT`/`TAIU1_OBJECT_HOUSE`: «Этот
   триггер вообще не нужен». Их стоит читать перед тем как опираться на такой код как на образец.

## История репозитория: более широкий контекст

В git-истории (не в текущем рабочем дереве) есть удалённые файлы: `README.md` (описывал репозиторий
как «Перенос пакета с процедурами и функциями из Oracle 10g в PostgreSQL 16»), `DDL_for_test` /
`DDL.sql` (тестовые `CREATE TABLE` + `INSERT` для минимального набора таблиц) и
`пакет_PLpgSQL.sql` — начатый порт `object_info_pkg` на PL/pgSQL для PostgreSQL (с построчными
комментариями-аналогиями Oracle→Postgres: `TTableNumber()` → `int8[]`, `.EXTEND` →
`array_append`, `CONNECT BY` → `WITH RECURSIVE` и т.д.). Все эти файлы удалены из текущего HEAD, но
их существование говорит о вероятной конечной цели работы с репозиторием — миграции этой Oracle-логики
на PostgreSQL 16. Если задача связана с портированием — стоит уточнить у пользователя, нужно ли
восстановить/продолжить эту ветку работы, а не считать её отменённой.

## Как вносятся изменения (по конвенциям кода)

- При изменении набора полей `OBJECT`, которые должны попадать в `object_info_tbl`:
  1. добавить колонку в `object_info_tbl` (`ALTER TABLE`, по образцу `00_`/`02_`-файлов);
  2. добавить поле в `l$object_required_column_list` в `object_info_pkg` (используется и для
     генерации триггера, и для копирования данных);
  3. обработать поле в обеих ветках `update_object_info(p$object OBJECT%ROWTYPE, p$action)` —
     `INSERT` и `UPDATE`;
  4. вызвать `object_info_pkg.create_triggers()`, чтобы перегенерировать триггер на `OBJECT`;
  5. при необходимости — `object_info_pkg.create_view()` для view;
  6. прогнать `rebuild()` / `rebuild(p$batch_size)` для уже существующих данных;
  7. поднять номер версии и дописать changelog в `get_version()`.
- При изменении логики наследования адреса между домом/подъездом/помещением/комнатой — править
  соответствующий `trigger.object_*.sql`, соблюдая правило `addressing_mode` и помня, что каскадные
  `UPDATE` на **другие** подтиповые таблицы допустимы только на уровне STATEMENT-триггера (через
  `num_collect_pkg`), а не прямо в теле `BEFORE ... FOR EACH ROW` (иначе — mutating table).
