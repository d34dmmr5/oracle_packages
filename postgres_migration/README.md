# Миграция BA7_DATA / object_info на PostgreSQL 16 — финальная (merge) версия

Порт Oracle-логики учёта объектов недвижимости (см. корень репозитория и `../CLAUDE.md`) на
PostgreSQL 16. Эта версия — **объединение двух независимых попыток порта** («лучшее из обоих»):
за основу взят вариант на transition tables (более полный по функциям и с тест-сьютом), поверх
наложены доработки надёжности из второго варианта. Все места доработок помечены в коде меткой
`MERGE-фикс`.

## Порядок разворачивания

```bash
# CREATE DATABASE выполнить отдельно (нельзя внутри транзакции):
#   createdb object_info_test
psql -d object_info_test -v ON_ERROR_STOP=1 -f 00_ddl.sql
psql -d object_info_test -v ON_ERROR_STOP=1 -f 01_territory_pkg.sql
psql -d object_info_test -v ON_ERROR_STOP=1 -f 02_object_info_pkg.sql
psql -d object_info_test -v ON_ERROR_STOP=1 -f 03_triggers.sql
psql -d object_info_test -v ON_ERROR_STOP=1 -f 04_test_data.sql   # тестовые данные (идемпотентно)
psql -d object_info_test                    -f 05_tests.sql        # T1–T16
```

Проверено на чистой PostgreSQL 16: все файлы применяются по порядку, T1–T16 проходят.

## Архитектура (общая для обоих исходных вариантов)

- **Схемы-«пакеты»**: код в `territory_pkg` и `object_info_pkg`, данные — в `public` (перенос в
  выделённую схему `ba7_data` — тривиальный последующий шаг, ортогональный логике; здесь оставлен
  `public`, чтобы не переписывать проверенный тест-сьют).
- **Составные типы вместо `%ROWTYPE`**: `object_info_pkg.object_row_type` (аналог `OBJECT%ROWTYPE`,
  только нужные адресу поля) и `public.territory_info_type` (результат сборки адреса).
- **Нет пакетных переменных**: вместо Oracle-коллекций, накапливаемых построчными триггерами и
  обрабатываемых на STATEMENT-уровне, — нативные **AFTER STATEMENT-триггеры с transition tables**
  (`REFERENCING NEW/OLD TABLE`): каждая DML-операция получает весь набор изменённых строк разом.
  На `object` — три отдельных триггера (INSERT/UPDATE/DELETE), т.к. один триггер с transition table
  обслуживает одно событие.
- **`num_collect_pkg` (обход mutating table)** для каскада «подъезд переехал в другое здание»
  заменён на AFTER STATEMENT-триггер с transition table на `house_doorway`.

## Что взято из какого варианта

**Из «transition»-варианта (база):**
- Полная Oracle-совместимая сигнатура `territory_pkg.get_info` (40 INOUT-параметров) + обёртка
  `get_territory_info`, возвращающая составной тип.
- **Управление триггерами на сессию** через GUC `app.trigger_taiud_e52_object_info` (порт Oracle
  `SYS_CONTEXT('CTX_TRIGGER_CONTROL', ...)`) — можно отключать пересчёт на время массовых операций.
- **Пропуск слияния объектов**: `app.writer = 'dublicate_pkg'` (порт Oracle-контекста `writer`).
- `object_info_pkg.get_version()`, `object_info_pkg.create_view()`.
- Комплект поведенческих тестов (T1–T12) с эталонными ожиданиями по Oracle-поведению.

**Из «reliability»-варианта (доработки, метка `MERGE-фикс`):**
- Защита от цикла в иерархии `territory` в обоих рекурсивных обходах.
- `COMMIT` по батчам в `rebuild(p_batch_size)`.
- Устойчивость к NULL/битым данным в сборке адреса.
- `CHECK (addressing_mode IN (1,2))`.
- Исправление двух багов Oracle-оригинала (см. ниже).

## Доработки надёжности (метка `MERGE-фикс` в коде)

1. **Защита от цикла в иерархии `territory`** — самый важный фикс. Oracle `CONNECT BY` сам ловит
   цикл в данных (`ORA-01436`); у `WITH RECURSIVE` такой защиты по умолчанию нет, и циклическая
   иерархия (`id_parent`: A→B→A — ошибка ввода оператором) уводит запрос в **настоящий бесконечный
   цикл, вешающий backend**. Уязвимы были **оба** рекурсивных обхода: восходящий в
   `territory_pkg.get_info` (сборка адреса) и нисходящий в `object_info_pkg.update_object_info_
   territories` (пересчёт потомков; вызывается триггером на `territory`, поэтому вешал даже обычный
   `UPDATE territory`). Оба переведены на нативную конструкцию PostgreSQL 14+
   `CYCLE ... SET ... USING ...`: рекурсия гарантированно конечна, при обнаружении цикла — `WARNING`
   и частично собранный результат вместо зависания. Проверено (T13): на цикле раньше — зависание на
   минуты с блокировкой таблицы, теперь — миллисекунды + предупреждение.
2. **`COMMIT` по батчам в `rebuild(p_batch_size)`**. Базовая batch-версия держала **одну
   транзакцию** на весь пересчёт (для 500K `object` — ~5 минут: огромный WAL, долгие блокировки, при
   сбое посреди откатывается всё). Теперь `COMMIT` после каждого батча: прогресс сохраняется,
   блокировки освобождаются, повторный запуск идемпотентно продолжает. **Для первичной загрузки
   больших объёмов используйте `rebuild(p_batch_size)`, а не голый `rebuild()`.**
3. **Устойчивость к NULL / «грязным» данным** в `update_object_info`. В Oracle `||` трактует `NULL`
   как пустую строку, в PostgreSQL — «заражает» `NULL`-ом всю строку. Без защиты одно пустое поле
   (дом без номера — обычный ручной ввод; или осиротевшая ссылка на территорию при заливке с
   отключёнными FK) обнуляло **весь** адрес, включая уже собранные страну/регион/город. Обёрнуто в
   `COALESCE`; при осиротевшей территории — ещё и `WARNING`. Проверено T14 (NULL-дом → частичный
   адрес), T15 (несуществующая территория → `WARNING` + частичный адрес).
4. **`CHECK (addressing_mode IN (1,2))`** на `object_house` — defense-in-depth. Триггер и так
   отклоняет иной режим (его ветка ELSE срабатывает раньше CHECK), но CHECK страхует на случай
   правки/отключения триггера. Проверено T16.

## Исправленные баги Oracle-оригинала (метка `MERGE-фикс`)

Оба сохранены «намеренно» в базовом варианте ради бит-в-бит совместимости; здесь **исправлены**, т.к.
задача была «взять лучшее». Если downstream-потребители завязаны на Oracle-выхлоп — в коде рядом с
каждым фиксом указано, как вернуть исходное поведение.

- **Дублирование `building_no2`**: в альтернативном адресе `корп. <N>` добавлялся дважды
  (`'корп. Б корп. Б'`). Исправлено на однократное — как в основном адресе. Видно в кросс-сверке
  (единственное расхождение с Oracle-верным вариантом на тестовых данных).
- **`id_street2` в UPDATE-ветке**: писался как `id_territory2` (лист иерархии), тогда как
  INSERT-ветка корректно пишет `id_street` (id улицы). Приведено к поведению INSERT. На обычных
  данных незаметно (альт-адрес вводят на уровне улицы, тогда `id_territory2 = id_street`), но при
  альт-территории не-улице расходится: старый код писал id не-улицы в поле улицы.

## Кросс-сверка с Oracle-верным вариантом (одинаковые данные)

Идентичный `04_test_data` прогнан через финальный код и через базовый (bug-for-bug верный Oracle)
вариант. Полный дамп адресов (`adres`, `adres2`, `id_street2`) совпал **везде, кроме двух намеренных
фиксов**:
- object 121 `adres2`: `корп. Б корп. Б` (старое) → `корп. Б` (финал);
- object 121 `id_street2` при альт-территории-городе: `4` (старое, id города) → `NULL` (финал,
  корректно — у города нет улицы).

То есть миграция сохраняет всё Oracle-поведение и меняет ровно то, что заявлено.

> Примечание про «прогон на Oracle»: сам Oracle в этом окружении не исполним (нет инстанса; Oracle-код
> опирается на внешние фреймворковые пакеты `scheming_pkg`/`ddl_pkg`/`int_rep_session`/…, которых нет
> в репозитории). Поэтому эталон Oracle-поведения — это (а) ожидаемые значения в T1–T16, выведенные из
> Oracle-логики, и (б) bug-for-bug верный базовый PostgreSQL-вариант как исполнимый прокси.

## Тесты (нет CI/фреймворка — проверка ручная, `05_tests.sql`)

- T1–T9 — базовая логика: смена номеров, каскады `territory` (переименование улицы/города вверх по
  дереву), quirk с `zip` (UPDATE только zip не пересчитывает), перенос подъезда между зданиями,
  каскад номера дома в режиме «по подъездам», удаление, отключение триггера через GUC, пропуск
  `writer='dublicate_pkg'`.
- T10–T11 — негативные: подъезд с номером при режиме 1, квартира без подъезда при режиме 2.
- T12 — `rebuild(5000)`: полное восстановление кэша с нуля (objects = cached = 13).
- **T13–T16 (стресс, MERGE)** — цикл в `territory` (не виснет), NULL-номер дома (частичный адрес),
  осиротевшая территория (`WARNING` + частичный адрес), кривой `addressing_mode` (отклонён).
